# Product Requirements & Testing Document: FinResilience Pro

## 1. Product Overview & Purpose

**FinResilience Pro** is an "engine-first" personal wealth orchestration platform tailored for the Indian financial ecosystem. It acts as a deterministic financial concierge. Instead of relying on generative AI to calculate math, the platform uses a strict Python backend to process user income, debts, and expenses. The frontend's primary responsibility is to securely capture user inputs, render the deterministic financial outputs (Action Cards, Payoff Schedules, Health Scores), and gracefully display contextual AI narratives.

**Primary Goal:** To tell Indian salaried individuals exactly what to do with their "next rupee" based on the Debt Avalanche method, Indian Statutory Tax rules (FY 2026-27), and commercial banking FOIR (Fixed Obligation to Income Ratio) benchmarks.

---

## 2. Frontend Tech Stack & Architecture

TestSpirit should generate tests assuming the following client-side environment:

* **Framework:** React 18 with TypeScript (Bootstrapped via Vite).
* **Styling:** Tailwind CSS.
* **Design System:** Dark fintech ambient gradients. **Strictly no glassmorphism**; UI relies on solid, elevated cards with subtle borders.
* **State Management:** Zustand (for global client state).
* **Data Visualization:** Recharts (for history and charts) and Vaul (for mobile-responsive drawers/modals).
* **Document Generation:** `@react-pdf/renderer` for client-side PDF export.

---

## 3. Core UI Components & Expected Behaviors

### 3.1. The "Next Rupee" Action Cards (Hero Section)

* **Description:** The highest priority visual element on the dashboard. These cards instruct the user on immediate financial actions.
* **Expected Behavior:**
* Must dynamically render state tags such as `PAY_DEBT`, `INVEST_TAX`, or `SAVE`.
* Must display the exact monetary value and the projected outcome (e.g., "Net savings of ₹12,340").



### 3.2. Month-by-Month Payoff Schedule

* **Description:** A deterministic ledger generated by the backend's Avalanche engine.
* **Expected Behavior:**
* Renders as a data table or list.
* Must be expandable/collapsible (accordion style) to show month-by-month debt reduction.
* Sorting and pagination (if applicable) must function without breaking layout.



### 3.3. Financial Health Gauges

* **Description:** Visual indicators representing the user's financial health based on Indian banking benchmarks.
* **Expected Behavior:**
* **FOIR Gauge:** Must reflect a 0-100% scale (Optimal is $\le$ 40-45%).
* **Savings Rate Gauge:** Target $\ge$ 20%.
* **Emergency Buffer Gauge:** Target $\ge$ 3 months.
* Gauges should animate smoothly on initial load (respecting user motion preferences).



### 3.4. AI Narrative Section

* **Description:** A text block at the bottom of the dashboard containing a 2-4 sentence contextual summary generated by Gemini 2.5 Flash.
* **Expected Behavior:**
* Must render markdown/formatted text correctly.
* **Offline/Fallback handling:** If the AI API fails or times out, the component must not crash the page. It must render a deterministic fallback text string provided by the local engine.



### 3.5. PDF Report Export

* **Description:** A client-side PDF generator for the user's financial snapshot.
* **Expected Behavior:**
* Clicking the "Export Report" button must trigger a browser download prompt.
* The process must happen securely on the client side without routing sensitive DOM data back to an external server.

## 4. Critical User Flows for Testing

1. **User Onboarding / Data Entry Flow:**
* User inputs unstructured income, monthly expenses, existing debts (balance, APR, EMI), and current tax regime investments.
* *Assertion:* Form inputs must validate standard currency formatting and prevent negative/invalid numbers.


2. **Dashboard Rendering Flow:**
* Upon successful data submission, the UI must transition to the `ResultsPanel`.
* *Assertion:* Action Cards, Schedule, Gauges, and AI Narrative must render in the correct visual hierarchy.


3. **Error Handling & Fallback Flow:**
* Simulate a network timeout when fetching the AI narrative.
* *Assertion:* The rest of the dashboard (Calculated Engine data) must remain fully visible and functional, displaying fallback text in the narrative block.

## 5. Accessibility (a11y) & UX Requirements

TestSpirit must validate the following compliance standards:

* **WCAG 2.1 AA Compliance:** High contrast ratios for text against dark fintech backgrounds.
* **Screen Reader Support:** All charts (Recharts) must have visually hidden table fallbacks or appropriate `aria-labels`.
* **ARIA Live Regions:** Dynamic updates (like goal progression or AI text generation completion) must use `aria-live="polite"` to notify assistive technologies.
* **Motion Control:** All CSS transitions and chart animations must be disabled or simplified if the user's OS has `prefers-reduced-motion` enabled.
